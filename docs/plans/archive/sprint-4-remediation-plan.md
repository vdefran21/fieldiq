## Context

A code review of the Sprint 4 negotiation protocol identified 6 bugs, all verified as accurate. The root causes:

1. **Relay responses are fire-and-forget** ‚Äî the initiator never reads the RelayResponse, so it never learns about match/failure on the remote side. The integration test masks this with a `syncSession()` helper that manually copies state between instances.
2. **Confirmation is single-sided** ‚Äî `confirmAgreement()` immediately transitions to `confirmed` and creates the Event on the first call; the remote `handleIncomingConfirm()` only writes an audit log. Spec requires both managers to confirm before events are created.
3. **Join endpoint skips authorization** ‚Äî the only negotiation endpoint that doesn't call `authenticatedUserId()` or `TeamAccessGuard`.
4. **`/incoming` relay endpoint is a stub** ‚Äî returns hardcoded response, no shadow session created on Instance B, HMAC filter blocks bootstrap.
5. **Counter path is half-wired** ‚Äî outgoing respond sends counter slots, but the inbound handler discards them.
6. **TypeScript types drifted** ‚Äî `proposals` field missing, `ProposeRequest` declares a body the controller doesn't accept.

This plan fixes all 6 in dependency order and removes the `syncSession()` workaround from integration tests, which is the true validation that cross-instance state propagation works.

### Architecture clarification: where does `join` happen?

The spec says "POST /negotiations/:id/join (on Instance B)" (doc 04, line 20). However, this is describing the user's context ‚Äî Manager B is using Instance B's mobile app ‚Äî not the API target. The session entity lives on Instance A (which created it). Manager B's app calls **Instance A's** join endpoint via the deep-link URL (which encodes Instance A's base URL). After successful join, Instance A bootstraps a shadow session on Instance B via `/api/negotiate/incoming`.

This "join targets the session owner" model is what the current code implements and is the correct architecture. The spec doc should be clarified with a parenthetical: "POST /negotiations/:id/join (called on Instance A, initiated by Manager B from Instance B's app)". This doc update is included in Phase 2.

## Phase 2: Shadow Session + Incoming Endpoint (Bug 4)

**Required before Bugs 1 and 2 ‚Äî Instance B needs a local session for relay processing.**

### Design Decision: `/incoming` authentication

The HMAC filter requires a local session for key derivation, but `/incoming` bootstraps the session ‚Äî it can't exist yet. The invite token in the request body serves as a **bearer secret**: it's 36 bytes of cryptographic randomness generated by Instance A and shared only with session participants. Instance B cannot independently verify that the token originated from a specific session on Instance A ‚Äî it trusts it the same way the join endpoint does. Knowing the token is sufficient proof of access for the bootstrap call.

**Approach:** Exclude `/incoming` from the HMAC filter. The service layer accepts the invite token, derives a session key from it, and creates the shadow session. No additional cryptographic verification is performed on the token itself ‚Äî it is a bearer credential.

### Design Decision: Shadow session creation is a hard requirement of join

If `/incoming` fails (network error, Instance B down), the join **must fail**. The invite token is NOT consumed. Rationale: all subsequent `/relay` calls require a local session on Instance B for HMAC validation (the filter looks up the session at line 154). A successful join with a failed shadow session leaves Instance B permanently unable to process relays for this negotiation.

**Flow:**

1. `joinSession()` validates invite token
2. `joinSession()` calls `sendIncoming()` to Instance B ‚Äî **before consuming the token**
3. If `/incoming` succeeds ‚Üí consume token, derive key, save session locally
4. If `/incoming` fails ‚Üí throw exception, join fails, token stays valid for retry

## Phase 4: Dual Confirmation + Deferred Event Creation (Bug 2)

**Requires Phase 3 so the initiator knows about `pending_approval`.**

### Design Decision: Events created only after both confirm

The spec at line 52-53 is clear: "Both managers confirm ‚Üí status = 'confirmed' ‚Üí Event created in FieldIQ for each team." Event creation is a consequence of reaching `confirmed`, not of individual confirmation. Creating events on first confirmation produces one-sided scheduled games if the second manager never confirms or later cancels.

**Approach:** Each `confirmAgreement()` call sets the local side's confirmation flag and persists the agreed slot. Only when both flags are true does the session transition to `confirmed` and the local Event gets created. `handleIncomingConfirm()` sets the remote flag and also checks + creates the event if both are now true.

### Design Decision: Confirmation and event creation must be idempotent

`CrossInstanceRelayClient.sendRelay()` already retries failed requests, so `confirm` can arrive more than once even when the user only taps once. Repeated `confirmAgreement()` calls from the same manager are also possible. Event creation therefore needs an explicit guard.

**Approach:** Before creating a local Event, look up an existing event by `(teamId, negotiationId)`. If one exists, return/reuse it instead of inserting another row. The confirmation flag update should also short-circuit if that actor has already confirmed. This keeps confirmation safely retryable.
